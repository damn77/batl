# Phase 1.1: Bracket Generation and Seeding Persistence - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning

<domain>
## Phase Boundary

An organizer closes registration and generates the tournament draw — computing seeded bracket positions from rankings and persisting them as actual Bracket/Round/Match records in the database. Phase 1.1 is the bridge between the bracket generation APIs (Features 009, 010) and Phase 2 (tournament lifecycle progression), which needs real DB records to advance winners through the bracket.

Out of scope: match result entry, advancing winners (Phase 2), full tournament lifecycle (Phase 2).

</domain>

<decisions>
## Implementation Decisions

### Organizer workflow

- Bracket generation is a distinct step, separate from starting the tournament: organizer closes registration → generates draw → reviews bracket → starts tournament
- Registration close is an explicit organizer action ("Close Registration" button), not automatic
- Generating the draw requires registration to be closed first (prerequisite)
- An explicit "Generate Draw" button initiates generation (not automatic on registration close)
- After generation, organizer views the bracket and can make manual slot edits, then separately clicks "Start Tournament" (Phase 2)
- If a player withdraws after registration closes but before draw generation: generate with remaining active players (no blocking)
- If a player withdraws after draw generation but before tournament start: organizer can choose to regenerate the draw (delete and replace) OR leave it as-is, in which case that player's matches become automatic forfeits

### Player assignment & seeding

- Auto-seed from rankings (Feature 008): system pulls top N registered players by seeding score and places them in optimal bracket positions using the existing seeding placement service (Feature 010)
- Players with no seeding score in this category are treated as unseeded and randomly placed in non-seed slots
- After generation, organizer CAN manually swap any two bracket slots (no restrictions — seeded or unseeded)
- For doubles tournaments: organizer can choose the seeding method before generating — either the pair's combined seeding score (Feature 008 PAIR ranking) or the average of both individual players' scores

### Regeneration & editing

- Regenerate = delete and replace: all existing Bracket/Round/Match records for this tournament are deleted and new ones created from scratch
- Regeneration is available at any point between generation and Start (before IN_PROGRESS status)
- Bracket is locked once the tournament starts — no regeneration possible after IN_PROGRESS
- Manual slot swaps after generation use batch editing: organizer can make multiple swaps, then clicks "Save Draw" to persist them all at once (not immediate per-swap)
- If regenerated, any unsaved manual edits from the current session are discarded

### Frontend surface

- Embedded in the existing tournament management page (not a separate route) — a new section appears when the tournament format is KNOCKOUT
- Before bracket generated (registration closed): shows finalized registered player list + prominent "Generate Draw" button
- After bracket generated: shows the bracket visualization with per-slot dropdown selectors for swapping players, plus "Regenerate Draw" and "Save Draw" buttons
- For doubles seeding method selection: a simple option (radio/select) before the Generate Draw button allowing organizer to pick "Pair score" or "Average player scores"

### Claude's Discretion

- Layout and visual design of the bracket generation section within the tournament management page
- Confirmation dialog wording for "Regenerate Draw" (since it deletes the current draw)
- How BYE slots are handled in the slot-swap UI (likely just shown as non-swappable)
- API design for slot swap persistence (PATCH on match record vs. dedicated swap endpoint)

</decisions>

<specifics>
## Specific Ideas

- The bracket visualization after generation should reuse the existing KnockoutBracket component (Feature 011) with an added organizer editing mode
- Slot dropdowns should show player names clearly; for doubles show the pair name
- The "Save Draw" button should be prominent and visually distinct from "Regenerate Draw" to prevent accidental regeneration

</specifics>
