# Phase 6.1: Match Result Resubmission and Bracket Recalculation - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

When a match result is resubmitted, correctly handle bracket state across both main and consolation brackets. Non-organizers are blocked from changing the winner. Organizers can change the winner with full cascade-clearing of downstream matches and a verification popup when later stages have already been played. Score-only corrections (same winner) are allowed for all participants without cascade.

</domain>

<decisions>
## Implementation Decisions

### Winner change blocking
- Any non-organizer resubmission that changes the winner is **always blocked**, regardless of downstream state
- Backend rejects the request with a structured error (same pattern as ORGANIZER_LOCKED)
- Error code: `WINNER_CHANGE_NOT_ALLOWED` (or similar)
- Error message includes guidance: "Only an organizer can change the winner of a completed match. Contact the tournament organizer to correct this result."
- Non-organizers CAN still resubmit with the same winner to correct scores (existing behavior preserved)

### Cascade-clear behavior (organizer winner change)
- When organizer changes the winner of a main bracket match, **all downstream main bracket matches** where the old winner (or someone who advanced because of the old winner) appears are cascade-cleared: player slot nulled, result cleared, status reset to SCHEDULED
- The new winner is **auto-advanced** to the next match slot (same as first-time submission via advanceBracketSlot)
- Cross-bracket cascade: existing `clearConsolationRouting()` already handles removing the old loser from consolation and cascading through consolation rounds; `routeLoserToConsolation()` re-routes the new loser — existing code path in matchResultService (lines 199-228) should work with main bracket cascade added
- For consolation bracket winner changes: impacted downstream consolation matches (1 per round) are identified and cascade-cleared, same recursive pattern as `cascadeClearWinnerFromNextRounds`

### Verification popup (organizer-only)
- Two-step flow using a **dry-run flag** on the existing submit endpoint: `?dryRun=true`
- Dry-run returns impact info: `{impacted: {mainMatches: N, consolationMatches: M, affectedPlayers: [{name, ...}]}}`
- Frontend shows confirmation popup **only when downstream impact exists** (mainMatches + consolationMatches > 0)
- Popup content: "Changing the winner will clear X match(es) in the main bracket and Y match(es) in the consolation bracket. The following players/pairs will be affected: [list]. Proceed?"
- If dryRun returns 0 impacted matches, skip the popup and proceed directly
- On confirmation, frontend re-calls the same endpoint without dryRun to execute

### Score-only updates (same winner)
- Current behavior preserved: overwrite the result JSON, no cascade, no bracket recalculation
- Frontend detects same winner before calling the API — skips the dry-run call entirely (no extra round trip)
- No audit trail for score changes (deferred to future AUDIT-01 requirement)

### Claude's Discretion
- Exact error code naming for the non-organizer block
- Dry-run response shape details beyond the core fields
- How the cascade traversal is implemented (recursive vs iterative)
- Confirmation popup styling and layout within existing React Bootstrap patterns

</decisions>

<specifics>
## Specific Ideas

- The dry-run check should be lightweight — count impacted matches and collect player names, but not actually modify anything
- The main bracket cascade pattern should mirror `cascadeClearWinnerFromNextRounds` (consolationEligibilityService.js) which already does recursive clearing — adapt for main bracket context
- matchResultService.submitResult() is the single entry point for all result submissions — the non-organizer block and dry-run logic should live here (or in a new helper it calls)
- The verification popup reuses the existing Modal pattern from MatchResultModal — it's a confirmation step within the same modal flow, not a separate component

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `clearConsolationRouting()` (consolationEligibilityService.js): Already cascade-clears consolation slots recursively when a winner changes — pattern to replicate for main bracket
- `cascadeClearWinnerFromNextRounds()` (consolationEligibilityService.js): Recursive consolation cascade — same algorithm needed for main bracket downstream clearing
- `advanceBracketSlot()` (tournamentLifecycleService.js): Auto-advances winner to next round — reused after cascade-clear + new winner placement
- `matchResultService.submitResult()`: Already detects winner changes (lines 199-218) via previousResultJson comparison — extend with non-organizer block and main bracket cascade
- `MatchResultModal.jsx`: Existing result entry UI — add confirmation step for organizer winner changes

### Established Patterns
- Structured error throwing: `throwError(statusCode, code, message)` in matchResultService
- All bracket mutations happen inside a single `prisma.$transaction` for atomicity
- Winner detection: `parsedResult.winner === 'PLAYER1'` with `pair1Id ?? player1Id` nullish coalescing for doubles/singles
- Consolation cascade uses `findFirst` + `orderBy matchNumber asc` to find the next downstream match, then recurses

### Integration Points
- `matchResultService.submitResult()` is the single write path — all changes flow through here
- `tournamentController.js` calls submitResult — dry-run flag would be parsed here and passed through
- Frontend `MatchResultModal.jsx` calls the submit API — needs to add dry-run pre-check and confirmation step for organizer winner changes

</code_context>

<deferred>
## Deferred Ideas

- Result edit audit trail (AUDIT-01) — logging who changed what and when; future v1.2+ requirement
- Undo/revert capability for cascade-cleared matches — organizer would need to re-enter all cleared results manually for now

</deferred>
