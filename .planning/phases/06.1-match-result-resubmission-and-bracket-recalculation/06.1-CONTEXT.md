# Phase 6.1: Match Result Resubmission and Bracket Recalculation - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

When a match result is resubmitted, correctly handle bracket state across both main and consolation brackets. Non-organizers are blocked from changing the winner (frontend warning + backend rejection). Organizers can change the winner with full cascade-clearing of downstream matches and a verification popup when later stages have already been played. Score-only corrections (same winner) are allowed for all participants without cascade. BYE auto-advancement in consolation is verified and tested for correctness.

</domain>

<decisions>
## Implementation Decisions

### Winner change blocking (non-organizer)
- **Defense in depth**: frontend warning BEFORE API call + backend rejection as fallback
- Frontend detects winner is changing when form is edited — shows yellow/info alert inline in the modal: "Only a tournament organizer can change the winner of a completed match. You can update the score without changing the winner."
- Submit button disabled while the selected winner differs from the existing winner (for non-organizers)
- Backend rejects with structured error `WINNER_CHANGE_NOT_ALLOWED` (403) if a non-organizer request somehow reaches the API with a changed winner
- Non-organizers CAN freely change scores as long as the declared winner remains the same

### Result form pre-fill on resubmission
- When modal opens for a match with an existing result, pre-fill the form: winner radio pre-selected, scores pre-filled from current result
- Non-organizer sees the winner selection locked (radio buttons disabled or visually locked)
- Organizer can change anything (winner, scores)

### Cascade-clear behavior (organizer winner change)
- **Full recursive cascade**: when organizer changes the winner of a main bracket match, ALL downstream main bracket matches from that point are cascade-cleared — player slots nulled, results cleared, status reset to SCHEDULED
- The new winner is auto-advanced to the next match slot (via existing `advanceBracketSlot`)
- Cross-bracket cascade: existing `clearConsolationRouting()` handles removing the old loser from consolation; `routeLoserToConsolation()` re-routes the new loser
- For consolation bracket winner changes: same cascade pattern — impacted downstream consolation matches (1 per round) are identified and cascade-cleared recursively
- All cascade operations run inside the existing `prisma.$transaction` for atomicity

### Verification popup (organizer-only, both bracket types)
- Two-step flow using a **dry-run flag** on the existing submit endpoint: `?dryRun=true`
- **Impact detection**: if ANY match downstream from the changed match has a result (status COMPLETED), show the confirmation popup
- If dry-run returns 0 impacted matches (no downstream results exist), skip popup and proceed directly
- **Popup content**: match count + player names — "Changing the winner will clear X main bracket match(es) and Y consolation match(es). Affected players: [list]. Proceed?"
- **Same popup for consolation winner changes** — consistent behavior regardless of bracket type
- Reuses the existing `pendingInvalidSubmit` two-phase state pattern from MatchResultModal (set state on dry-run, show confirmation, re-call on confirm)
- On confirmation, frontend re-calls the same endpoint without dryRun to execute

### Score-only updates (same winner)
- Current behavior preserved: overwrite the result JSON, no cascade, no bracket recalculation
- Frontend detects same winner before calling the API — skips the dry-run call entirely (no extra round trip)
- No audit trail for score changes (deferred to future AUDIT-01 requirement)

### BYE auto-advancement in consolation
- Existing Phase 5 code (`routeLoserToConsolation`, `checkBYEWinnerConsolationUpdate`) handles BYE auto-advancement — this phase verifies and fixes, not rewrites
- P1/P2 assignment follows mirror main bracket order: loser from lower-numbered main match = P1, loser from higher-numbered main match = P2
- BYE-origin players (main bracket BYE in R1 who lose in R2) have played 1 real match — eligible for consolation under MATCH_2. No special treatment.
- **Dedicated test coverage**: write specific integration tests for all BYE scenarios from criterion 6, fix anything that fails

### Claude's Discretion
- Exact error code naming for the non-organizer block
- Dry-run response shape details beyond the core fields (match count, player names)
- How the cascade traversal is implemented (recursive vs iterative)
- Confirmation popup styling and layout within existing React Bootstrap patterns
- How the winner-locked UI state is visually communicated to non-organizers

</decisions>

<specifics>
## Specific Ideas

- The dry-run check should be lightweight — count impacted matches and collect player names, but not actually modify anything
- The main bracket cascade pattern should mirror `cascadeClearWinnerFromNextRounds` (consolationEligibilityService.js) which already does recursive clearing — adapt for main bracket context
- `matchResultService.submitResult()` is the single entry point for all result submissions — the non-organizer block and dry-run logic should live here (or in a new helper it calls)
- The verification popup reuses the existing `pendingInvalidSubmit` pattern from MatchResultModal — it's a confirmation step within the same modal flow, not a separate component
- Frontend winner-change detection: compare the selected winner against the pre-filled existing winner from `match.result` JSON

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `clearConsolationRouting()` (consolationEligibilityService.js): Already cascade-clears consolation slots recursively — pattern to replicate for main bracket
- `cascadeClearWinnerFromNextRounds()` (consolationEligibilityService.js): Recursive consolation cascade — same algorithm needed for main bracket downstream clearing
- `advanceBracketSlot()` (tournamentLifecycleService.js): Auto-advances winner to next round — reused after cascade-clear + new winner placement
- `matchResultService.submitResult()`: Already detects winner changes via previousResultJson comparison — extend with non-organizer block and main bracket cascade
- `MatchResultModal.jsx` `pendingInvalidSubmit` pattern: Two-phase confirmation flow (state holds pending data, conditional UI shows confirm/cancel) — exact pattern for winner-change confirmation
- `routeLoserToConsolation()` + `checkBYEWinnerConsolationUpdate()`: Existing BYE handling for consolation entry

### Established Patterns
- Structured error throwing: `throwError(statusCode, code, message)` in matchResultService
- All bracket mutations happen inside a single `prisma.$transaction` for atomicity
- Winner detection: `parsedResult.winner === 'PLAYER1'` with `pair1Id ?? player1Id` nullish coalescing for doubles/singles
- Consolation cascade uses `findFirst` + `orderBy matchNumber asc` to find the next downstream match, then recurses

### Integration Points
- `matchResultService.submitResult()` is the single write path — all changes flow through here
- `tournamentController.js` (or matchController) calls submitResult — dry-run flag parsed from `req.query.dryRun` and passed through
- Frontend `MatchResultModal.jsx` calls the submit API — needs dry-run pre-check and confirmation step for organizer winner changes

</code_context>
