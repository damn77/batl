## Cross-Table Cell States

All states must be visually distinct and implementation-ready:

| State | Visual | Cursor | Interaction |
|-------|--------|--------|-------------|
| Diagonal cell | `background: #f0f0f0`, no content | `default` | None ŌĆö no click, no hover highlight |
| Unplayed cell | Text: `ŌĆö` (en-dash), `text-muted` | `pointer` | Click opens MatchResultModal for result entry |
| Win cell | Score text (e.g., "2-0" or "6-4"), `text-success fw-bold` | `pointer` | Click opens MatchResultModal for viewing/editing |
| Loss cell | Score text (e.g., "0-2" or "4-6"), `text-muted` | `pointer` | Click opens MatchResultModal for viewing/editing |
| Cross-highlighted | `background: #e8f4fd` (on top of win/loss/unplayed) | unchanged | Desktop only, on hover of sibling match row |

Score format rules (from locked decisions):
- Multi-set matches: set count from row player's perspective ŌĆö e.g., "2-1"
- BIG_TIEBREAK / single-set (`scoringRules.winningSets === 1` or `formatType === 'BIG_TIEBREAK'`): game score ŌĆö e.g., "6-4"
- Unplayed: `ŌĆö` (en-dash, not empty string ŌĆö screen reader accessible)

## Cross-Highlighting Contract

Desktop only (Ōēź 576px). Shared `highlightedMatchId` state in `GroupStandingsTable`.

| Trigger | Effect |
|---------|--------|
| `mouseenter` on match list row | Apply `background: #e8f4fd` to both `[row, col]` and `[col, row]` cross-table cells for that matchId |
| `mouseleave` on match list row | Clear `highlightedMatchId` to `null` |
| `mouseenter` on cross-table cell | Apply `background: #e8f4fd` to the corresponding match list row |
| `mouseleave` on cross-table cell | Clear `highlightedMatchId` to `null` |
| Diagonal cell | Never participates in highlighting ŌĆö no matchId, no handlers |
| Tab change (`onSelect`) | Clear `highlightedMatchId` to `null` (prevents stale state) |

CSS class for highlight: `cross-highlight` with rule `background-color: #e8f4fd !important` (needs `!important` to override Bootstrap's striped table row background).

Mobile: No `onMouseEnter`/`onMouseLeave` handlers attached (touch devices don't fire these; avoid binding unnecessary events).

## Initials Format Contract

| Scenario | Format | Example | Collision handling |
|----------|--------|---------|-------------------|
| Singles player | First char of first name + first char of last name, uppercase | "Alice Liddell" ŌåÆ "AL" | Append numeric suffix: "AL", "AL2", "AL3" |
| Doubles pair | Singles initials of player 1 + "/" + singles initials of player 2 | "Alice L / Bob M" ŌåÆ "AL/BM" | Same ŌĆö suffix the colliding pair initials |
| Single-word name | First 2 chars uppercase | "Madonna" ŌåÆ "MA" | Same |
| Missing name | "?" | ŌĆö | ŌĆö |

Column header `<th>` uses initials only. Row `<th>` (Player column) shows full entity name as existing standings table does.

Source: CONTEXT.md (initials decision), RESEARCH.md (doubles initials format ŌĆö Claude's discretion resolved here).

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Cross-table keyboard navigation | `<table>` is focusable; `<td>` interactive cells use `tabIndex={0}` and respond to `Enter`/`Space` keypress opening modal |
