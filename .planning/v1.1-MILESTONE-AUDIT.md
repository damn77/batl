## Critical Gaps (Blockers)

### GAP-1 (High): LIFE-05 — Doubles post-placement opt-out places opponent in wrong slot

**Requirement:** LIFE-05 — "opt-out treated as automatic forfeit (opponent advances without playing)"

**Issue:** `consolationOptOutService.js` Step 7 calls `advanceBracketSlot(tx, currentMatch, opponentId)` with a SCHEDULED match that has no result JSON. Inside `advanceBracketSlot`, `winnerIsPlayer1` is derived from `resultJson?.winner === 'PLAYER1'` — which evaluates to `false` when `resultJson` is null. This always writes the advancing opponent into the pair2/player2 slot of the next round, regardless of which slot the opponent actually occupies in `currentMatch`. For approximately 50% of doubles opt-out cases this produces incorrect bracket placement.

**Files:** `backend/src/services/consolationOptOutService.js` (Step 7), `backend/src/services/tournamentLifecycleService.js` (`advanceBracketSlot`)

**Fix:** Determine slot direction from `currentMatch` slots directly (is `opponentId === currentMatch.pair1Id` → slot 1, else slot 2) and either write a synthetic result field or modify `advanceBracketSlot` to accept an explicit slot parameter for forfeit advancements.

---

### GAP-2 (High): LIFE-05 — No frontend UI for consolation opt-out

**Requirement:** LIFE-05 — "Player/pair can opt out of consolation participation at any time"

**Issue:** No frontend component, service function, page, or button calls `POST /api/v1/tournaments/:id/consolation-opt-out`. The endpoint is authenticated and fully implemented on the backend, but players and organizers can only trigger it via direct API calls.

**Fix required:** Add a frontend service function in an appropriate API service file, and a UI affordance (button or modal) on the tournament detail page or match result flow.

---

### GAP-3 (Medium): DRAW-02 / LIFE-03 — Doubles consolation progression: winnerId null

**Requirement:** DRAW-02 (slots populate), LIFE-03 (consolation winners advance)

**Issue:** `matchResultService.js` line 187: `const winnerId = parsedResult.winner === 'PLAYER1' ? updated.player1Id : updated.player2Id`. For doubles main bracket matches, `player1Id`/`player2Id` are null (entities stored in `pair1Id`/`pair2Id`). The null `winnerId` is passed to both `advanceBracketSlot` and `routeLoserToConsolation`. Initial loser routing recovers correctly (reads pair slots from `completedMatch` directly), but advancing consolation round winners after a result submission relies on `advanceBracketSlot` receiving a valid entity ID.

**Files:** `backend/src/services/matchResultService.js` line 187

**Fix:** Derive `winnerId` accounting for doubles — `const winnerId = parsedResult.winner === 'PLAYER1' ? (updated.pair1Id || updated.player1Id) : (updated.pair2Id || updated.player2Id)`.
