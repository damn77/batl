## User Constraints (from CONTEXT.md)

### Locked Decisions

**Organizer workflow:**
- Bracket generation is a distinct step, separate from starting the tournament: organizer closes registration → generates draw → reviews bracket → starts tournament
- Registration close is an explicit organizer action ("Close Registration" button), not automatic
- Generating the draw requires registration to be closed first (prerequisite)
- An explicit "Generate Draw" button initiates generation (not automatic on registration close)
- After generation, organizer views the bracket and can make manual slot edits, then separately clicks "Start Tournament" (Phase 2)
- If a player withdraws after registration closes but before draw generation: generate with remaining active players (no blocking)
- If a player withdraws after draw generation but before tournament start: organizer can choose to regenerate the draw (delete and replace) OR leave it as-is, in which case that player's matches become automatic forfeits

**Player assignment & seeding:**
- Auto-seed from rankings (Feature 008): system pulls top N registered players by seeding score and places them in optimal bracket positions using the existing seeding placement service (Feature 010)
- Players with no seeding score in this category are treated as unseeded and randomly placed in non-seed slots
- After generation, organizer CAN manually swap any two bracket slots (no restrictions — seeded or unseeded)
- For doubles tournaments: organizer can choose the seeding method before generating — either the pair's combined seeding score (Feature 008 PAIR ranking) or the average of both individual players' scores

**Regeneration & editing:**
- Regenerate = delete and replace: all existing Bracket/Round/Match records for this tournament are deleted and new ones created from scratch
- Regeneration is available at any point between generation and Start (before IN_PROGRESS status)
- Bracket is locked once the tournament starts — no regeneration possible after IN_PROGRESS
- Manual slot swaps after generation use batch editing: organizer can make multiple swaps, then clicks "Save Draw" to persist them all at once (not immediate per-swap)
- If regenerated, any unsaved manual edits from the current session are discarded

**Frontend surface:**
- Embedded in the existing tournament management page (not a separate route) — a new section appears when the tournament format is KNOCKOUT
- Before bracket generated (registration closed): shows finalized registered player list + prominent "Generate Draw" button
- After bracket generated: shows the bracket visualization with per-slot dropdown selectors for swapping players, plus "Regenerate Draw" and "Save Draw" buttons
- For doubles seeding method selection: a simple option (radio/select) before the Generate Draw button allowing organizer to pick "Pair score" or "Average player scores"

### Claude's Discretion
- Layout and visual design of the bracket generation section within the tournament management page
- Confirmation dialog wording for "Regenerate Draw" (since it deletes the current draw)
- How BYE slots are handled in the slot-swap UI (likely just shown as non-swappable)
- API design for slot swap persistence (PATCH on match record vs. dedicated swap endpoint)

## Summary

Phase 1.1 is the persistence bridge between two completed API layers (Features 009/010) and the Phase 2 lifecycle engine. The seeding placement service (`seedingPlacementService.js`) already computes positions in memory; the only missing piece is persisting those positions as `Bracket`, `Round`, and `Match` DB records via Prisma, plus the organizer UI to trigger and edit the draw.

The DB schema (`prisma/schema.prisma`) already contains all needed models: `Bracket`, `Round`, `Match`, `TournamentRegistration`, and `PairRegistration`. No schema migrations are required for the core persistence task. The `Tournament.status` field already has the `SCHEDULED`/`IN_PROGRESS` lifecycle values that gate bracket locking. The existing `TournamentViewPage` with `FormatVisualization` → `KnockoutBracket` already renders DB-stored matches, so once records are persisted the read path is free.

The two major work areas are: (1) a new backend service and API endpoints for "close registration + generate draw + regenerate draw + save slot swaps", and (2) a new frontend section (`BracketGenerationSection` component) embedded in `TournamentViewPage` shown only when `formatType === 'KNOCKOUT'` and the user is an organizer.

**Primary recommendation:** Build a `bracketPersistenceService.js` that wraps `generateSeededBracket()` from Feature 010, then maps the returned `positions` array into round-structured `Bracket`/`Round`/`Match` Prisma creates inside a single `$transaction`. Wire this to a new route `POST /api/v1/tournaments/:id/bracket`. For the slot swap, add `PATCH /api/v1/tournaments/:id/bracket/slots` (batch body). On the frontend, compose a `BracketGenerationSection` component embedded in `TournamentViewPage` using the existing `KnockoutBracket` in a new organizer-edit mode with per-slot dropdowns.

### Pattern 1: Single-Transaction Bracket Persistence

**What:** All Bracket, Round, and Match records for a tournament are created atomically. Any partial failure rolls back the entire transaction.

**When to use:** Whenever generating (or regenerating) a draw.

**How rounds map from positions:**
The `generateSeededBracket()` result returns a flat `positions` array (length = bracketSize). The bracket structure is a knockout tree. Rounds are derived as follows:
- bracketSize = 2^N means N rounds total
- Round 1 (preliminary) has bracketSize/2 match "slots" from the positions array
- Each subsequent round has half as many match slots
- Round 1 matches are created from adjacent pairs of positions: (pos[0], pos[1]), (pos[2], pos[3]), etc.
- Positions where `isBye === true` create a Match with `status: 'BYE'`, `isBye: true`, `player2Id: null`
- Positions where `isPreliminary === true` create a regular Match with `status: 'SCHEDULED'`

**Example (bracketSize = 16, 4 players registered → 11 player example from Feature 009):**
```
Round 1: 8 matches (from 16 positions, paired in groups of 2)
Round 2: 4 matches (quarterfinals)
Round 3: 2 matches (semifinals)
Round 4: 1 match  (final)
```

**Code pattern for the persistence service:**
```javascript
// Source: internal pattern, Prisma docs https://www.prisma.io/docs/orm/prisma-client/queries/transactions
export async function persistGeneratedBracket(tournamentId, positions, bracketSize, randomSeed) {
  return await prisma.$transaction(async (tx) => {
    // Delete existing bracket data (for regeneration)
    await tx.match.deleteMany({ where: { tournamentId } });
    await tx.round.deleteMany({ where: { tournamentId } });
    await tx.bracket.deleteMany({ where: { tournamentId } });

    // Create the Bracket record
    const bracket = await tx.bracket.create({
      data: {
        tournamentId,
        bracketType: 'MAIN',
        matchGuarantee: 'MATCH_1',
      }
    });

    const totalRounds = Math.log2(bracketSize);
    let matchNumber = 1;

    for (let roundNum = 1; roundNum <= totalRounds; roundNum++) {
      const round = await tx.round.create({
        data: {
          tournamentId,
          bracketId: bracket.id,
          roundNumber: roundNum,
        }
      });

      if (roundNum === 1) {
        // Create matches from pairs of positions
        const matchCount = bracketSize / 2;
        for (let i = 0; i < matchCount; i++) {
          const pos1 = positions[i * 2];
          const pos2 = positions[i * 2 + 1];
          const isByeMatch = pos1.isBye || pos2.isBye || !pos2.entityId;

          await tx.match.create({
            data: {
              tournamentId,
              bracketId: bracket.id,
              roundId: round.id,
              matchNumber: matchNumber++,
              player1Id: pos1.entityId,  // will be null for fully empty slots
              player2Id: pos2.entityId,
              isBye: isByeMatch,
              status: isByeMatch ? 'BYE' : 'SCHEDULED',
            }
          });
        }
      } else {
        // Create placeholder matches for later rounds (players TBD)
        const matchCount = bracketSize / Math.pow(2, roundNum);
        for (let i = 0; i < matchCount; i++) {
          await tx.match.create({
            data: {
              tournamentId,
              bracketId: bracket.id,
              roundId: round.id,
              matchNumber: matchNumber++,
              player1Id: /* TBD_PLACEHOLDER_PLAYER_ID or null */ null,
              player2Id: null,
              status: 'SCHEDULED',
            }
          });
        }
      }
    }

    return bracket;
  });
}
```

**CRITICAL constraint:** The `Match` model has `player1Id String` (non-optional) pointing to a `PlayerProfile`. Later-round matches don't have players yet. Two options:
1. Use a sentinel "TBD" PlayerProfile record
2. Make player1Id nullable in the schema (requires migration)

See Open Questions section for resolution.

### Pattern 2: Close Registration Action

**What:** Organizer closes registration — sets `Tournament.status` to a new state OR uses `registrationCloseDate` to signal registration is closed.

**Current schema gap:** There is no `REGISTRATION_CLOSED` tournament status. The existing `TournamentStatus` enum has: `SCHEDULED`, `IN_PROGRESS`, `COMPLETED`, `CANCELLED`.

**Recommended approach:** Add `REGISTRATION_CLOSED` enum value between SCHEDULED and IN_PROGRESS. Alternatively, use a boolean flag `registrationClosed: Boolean` on Tournament without changing the status enum.

See Open Questions for the resolution.

### Pattern 3: Slot Swap (Batch)

**What:** Organizer swaps two player slots in the bracket. This modifies `Match.player1Id` or `Match.player2Id` for two matches to exchange two players.

**API design (Claude's discretion):** A dedicated batch swap endpoint is cleaner than individual PATCH calls.

```
PATCH /api/v1/tournaments/:id/bracket/slots
Body: { swaps: [{ matchId1, position1, matchId2, position2 }, ...] }
```

Each swap entry specifies two match+position pairs and the service atomically exchanges the player IDs between them.

**Constraint:** BYE slots must not be included in swaps. The service validates that neither position is a BYE match.

### Pattern 4: Doubles Seeding Method Selection

**What:** Before generating, organizer selects "Pair score" or "Average player scores". This affects how the top-N seeds are computed.

**Pair score path:** Use existing `POST /api/v1/seeding/generate-bracket` which already pulls `RankingEntry.seedingScore` for PAIR entities.

**Average player scores path:** Requires fetching each player's individual `RankingEntry.seedingScore` and averaging for the pair. This is new logic — it lives in `bracketPersistenceService.js` before calling `generateSeededBracket()`.

### Anti-Patterns to Avoid

- **Saving slot swaps immediately on each change:** The user decided batch saving (Save Draw button). Do not wire individual dropdowns to API calls — collect in local React state and flush on Save.
- **Creating matches without a Bracket record:** The `KnockoutBracket` component expects `matches` filtered by `bracketId`. Bracket must be created first.
- **Leaving later-round matches with nullable player1Id:** The current `Match` schema requires `player1Id`. Either make it nullable (schema change) or use a sentinel. See Open Questions.
- **Using the existing `POST /api/v1/seeding/generate-bracket` endpoint directly for persistence:** That endpoint generates in-memory only and returns JSON. Phase 1.1 needs a new endpoint that generates + persists.
- **Deleting bracket records outside a transaction:** Orphaned Round/Match records would break FK constraints in partial failure.

## Common Pitfalls

### Pitfall 1: Match.player1Id is Non-Nullable

**What goes wrong:** Creating later-round matches (Round 2, 3, 4) fails because `player1Id` is `String` (non-nullable) in the Prisma schema, but the players for those rounds aren't known yet.

**Why it happens:** The schema was designed for phases where matches already have players. Phase 1.1 pre-creates all match slots including empty future-round ones.

**How to avoid:** Two options — choose one before implementation:
1. Add a migration to make `player1Id` and `player2Id` both optional (`String?`). This is the cleanest approach and aligns with how BYE matches already have `player2Id` as nullable.
2. Use a well-known sentinel player ID (e.g., a special `PlayerProfile` created by seed data with `name: "TBD"`). Less clean but avoids a migration.

**Warning signs:** Prisma `create` call fails with required field constraint when creating Round 2+ matches.

### Pitfall 2: Regeneration Leaves Orphaned Records

**What goes wrong:** Regeneration deletes Bracket records but the cascade isn't in the right order — Match records reference both `bracketId` and `roundId`, so deleting Bracket without deleting Matches first violates FK constraints.

**Why it happens:** The cascade order matters: Matches first, then Rounds, then Brackets.

**How to avoid:** Always delete in this order inside the transaction:
```javascript
await tx.match.deleteMany({ where: { tournamentId } });
await tx.round.deleteMany({ where: { tournamentId } });
await tx.bracket.deleteMany({ where: { tournamentId } });
```

**Warning signs:** Prisma FK constraint error on `bracketId` or `roundId` during deleteMany.

### Pitfall 3: FormatVisualization Shows Empty After Generation

**What goes wrong:** After the organizer generates the draw and the frontend invalidates SWR cache, the `FormatVisualization` component still shows "not configured" because it uses a separate SWR key (`/tournaments/:id/format-structure`) with a 5-minute dedup interval.

**Why it happens:** `useFormatStructure` has `dedupingInterval: 300000` (5 min). It won't re-fetch after mutation unless `mutate()` is called explicitly.

**How to avoid:** After the generate-draw API call succeeds, explicitly call `mutate()` on the `useFormatStructure` SWR key to force a refresh. The `BracketGenerationSection` component must receive the `mutate` ref from the parent or manage its own SWR instance.

**Warning signs:** Bracket renders correctly on page reload but not immediately after generation.

### Pitfall 4: KnockoutBracket Expects Specific Match Structure

**What goes wrong:** The `KnockoutBracket` component reads `match.player1`, `match.player2` (joined objects with `id`, `name`), `match.isBye`, `match.status`, and `match.roundId`. If the backend's `getMatches()` service doesn't include the player join, the bracket renders blank slots.

**Why it happens:** The existing `getMatches()` in `tournamentService.js` only includes `player1` and `player2` selects — it does not include `pair1`, `pair2`. For doubles tournaments, the slot-swap UI also needs pair data.

**How to avoid:** Extend `getMatches()` to include `pair1` and `pair2` with their players for doubles tournaments, or add a separate query path. For singles, the existing join is sufficient.

**Warning signs:** Bracket visualizes with "TBD" for all slots even after generation.

### Pitfall 5: "Close Registration" Status Gap

**What goes wrong:** The current `TournamentStatus` enum has no state for "registration closed, draw not yet generated". Using `SCHEDULED` for both "open for registration" and "registration closed" makes it impossible to enforce the prerequisite ("registration must be closed before generating draw").

**Why it happens:** The original schema was designed before Phase 1.1 was added.

**How to avoid:** Either (a) add `REGISTRATION_CLOSED` enum value to the Prisma schema and run a migration, or (b) add a boolean field `registrationClosed: Boolean @default(false)` to `Tournament` without touching the status enum. Option (b) avoids adding a new status the Phase 2 lifecycle must handle. Recommend option (b).

**Warning signs:** Organizer can click "Generate Draw" on a tournament that still has open registration.

### Pitfall 6: Doubles Average-Score Seeding Needs Category Context

**What goes wrong:** To compute average player scores for a doubles pair, we need each player's `RankingEntry.seedingScore` in this specific category/year. The `getSeededPlayers()` function in `seedingPlacementService.js` uses `getRankingsForCategory()` — but for doubles we may need to filter to the individual (MEN/WOMEN) ranking type, not the PAIR ranking type.

**Why it happens:** `getRankingsForCategory()` returns all ranking types (PAIR, MEN, WOMEN). Flat merging may mix PAIR and individual entries.

**How to avoid:** For the "average player scores" doubles seeding path, query `RankingEntry` specifically for `entityType: PLAYER` in the current year for that category, then for each pair look up both players' entries and average their `seedingScore`.

## Open Questions

1. **How to handle future-round Match records with no players yet**
   - What we know: `Match.player1Id` is `String` (non-nullable) in the Prisma schema. Later-round matches (semifinals, finals) don't have players assigned at draw time.
   - What's unclear: Is it better to make `player1Id` nullable (schema migration) or use a sentinel "TBD" player?
   - Recommendation: Make `player1Id` nullable (`String?`) with a migration. This is consistent with `player2Id` already being nullable (for BYE matches). One migration: `ALTER TABLE "Match" ALTER COLUMN "player1Id" DROP NOT NULL`.

2. **How to represent "registration closed" state**
   - What we know: The current `TournamentStatus` enum doesn't have a REGISTRATION_CLOSED value.
   - What's unclear: Adding a new enum value means Phase 2 lifecycle transitions need to account for it. Adding a boolean flag avoids touching the enum.
   - Recommendation: Add `registrationClosed: Boolean @default(false)` to Tournament. This avoids a status enum migration and Phase 2's `LIFE-01` ("start tournament") can check `registrationClosed === true` as a pre-condition. The "Close Registration" button simply PATCHes `registrationClosed = true`.

3. **API design for slot swaps**
   - What we know: The user decided batch saves (not per-swap). Claude has discretion on the API design.
   - What's unclear: Single endpoint vs. generic match update.
   - Recommendation: `PATCH /api/v1/tournaments/:id/bracket/slots` with body `{ swaps: [{ matchId, field: 'player1Id'|'player2Id', newPlayerId }] }`. Simpler than a two-slot-swap atomic endpoint; the batch is just N updates inside a transaction. BYE slot guard: validate none of the targeted matches have `isBye: true`.

4. **Does the doubles "average player score" path need to handle players with no ranking?**
   - What we know: Players with no seeding score are treated as unseeded (score = 0). The decision says "players with no seeding score are treated as unseeded and randomly placed in non-seed slots."
   - What's unclear: For doubles average scoring — if one partner has a score and the other doesn't, average = score/2 vs. treat the pair as unseeded.
   - Recommendation: Average their individual scores (score or 0 if no entry). Sort pairs by descending average to determine seeding order. Pairs below the seed count threshold are unseeded regardless of average.
