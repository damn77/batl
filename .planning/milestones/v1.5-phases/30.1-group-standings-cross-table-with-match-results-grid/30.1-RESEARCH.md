## Summary

Phase 30.1 is a pure frontend extension of `GroupStandingsTable.jsx`. It adds three features: (1) a cross-table (matrix grid) derived from existing match data showing head-to-head scores, (2) tabbed navigation wrapping the cross-table, existing standings table, and match list with responsive 2-tab / 3-tab switching, and (3) bidirectional cross-highlighting between cross-table cells and match list rows using shared hover state.

No backend changes are needed. All data is already available via `useMatches()` and `useGroupStandings()` SWR hooks. The cross-table is a pure presentation layer derived from the `matches` array: build an `entityId → entityId → match` lookup map, then render an N×N grid.

The main architectural challenge is the responsive tab switching. The project's established breakpoint is `575.98px` (sm). On desktop (≥576px) the match list lives below the tabs; on mobile (<576px) the match list becomes its own third tab. This requires either a `useWindowSize`/`useMediaQuery` hook or a CSS-only approach that renders both layouts and hides the inactive one. A CSS media query approach is simpler and avoids hydration/resize jank.

**Primary recommendation:** Extend `GroupStandingsTable.jsx` by adding (a) a `CrossTable` sub-component that receives `entities`, `matches`, and hover/highlight state, (b) React Bootstrap `Tab.Container` / `Nav` / `Tab.Pane` tab wrapper, and (c) a shared `highlightedMatchId` state in `GroupStandingsTable` for cross-highlighting. No new SWR hooks required.

### Pattern 1: Match Lookup Map (cross-table data derivation)

**What:** Build a 2D lookup object from `matches` array so cell rendering is O(1).

**When to use:** Computed once with `useMemo`, recomputed when `matches` changes.

```javascript
// Derived in GroupStandingsTable (or CrossTable) via useMemo
const matchLookup = useMemo(() => {
  const map = {};
  if (!matches) return map;
  for (const match of matches) {
    const p1Id = match.player1Id || match.pair1Id;
    const p2Id = match.player2Id || match.pair2Id;
    if (p1Id && p2Id) {
      if (!map[p1Id]) map[p1Id] = {};
      if (!map[p2Id]) map[p2Id] = {};
      map[p1Id][p2Id] = match;
      map[p2Id][p1Id] = match; // symmetric lookup — both directions
    }
  }
  return map;
}, [matches]);
```

Note: `match.player1Id` / `match.player2Id` vs `match.pair1Id` / `match.pair2Id` depends on `isDoubles`. The match object from the API includes `player1`, `player2`, `pair1`, `pair2` as nested objects — extract IDs from those (e.g., `match.player1?.id`).

### Pattern 2: Cell Score Rendering

**What:** Compute cell content (score string) from the match's `result` JSON and `scoringRules`.

**When to use:** Called per cell in the cross-table render.

```javascript
function getCellScore(match, rowEntityId, scoringRules) {
  if (!match || match.status !== 'COMPLETED') return null;
  const result = typeof match.result === 'string'
    ? (() => { try { return JSON.parse(match.result); } catch { return null; } })()
    : match.result;
  if (!result) return null;

  const p1Id = match.player1?.id || match.pair1?.id;
  const isRowPlayer1 = (rowEntityId === p1Id);

  // BIG_TIEBREAK / single-set: show game score (6-4), not set count (1-0)
  const isSingleSet = scoringRules?.formatType === 'BIG_TIEBREAK'
    || (scoringRules?.winningSets === 1);

  if (isSingleSet) {
    const set = result.sets?.[0];
    if (!set) return null;
    const s1 = isRowPlayer1 ? set.player1Score : set.player2Score;
    const s2 = isRowPlayer1 ? set.player2Score : set.player1Score;
    return `${s1}-${s2}`;
  }

  // Multi-set: count sets won from row player's perspective
  const sets = result.sets || [];
  let rowWins = 0, oppWins = 0;
  for (const s of sets) {
    const rowScore = isRowPlayer1 ? s.player1Score : s.player2Score;
    const oppScore = isRowPlayer1 ? s.player2Score : s.player1Score;
    if (rowScore > oppScore) rowWins++; else oppWins++;
  }
  return `${rowWins}-${oppWins}`;
}

function getCellWin(match, rowEntityId) {
  if (!match || match.status !== 'COMPLETED') return null;
  const result = typeof match.result === 'string'
    ? (() => { try { return JSON.parse(match.result); } catch { return null; } })()
    : match.result;
  if (!result?.winner) return null;
  const p1Id = match.player1?.id || match.pair1?.id;
  const isRowPlayer1 = (rowEntityId === p1Id);
  if (result.winner === 'PLAYER1') return isRowPlayer1 ? 'win' : 'loss';
  if (result.winner === 'PLAYER2') return isRowPlayer1 ? 'loss' : 'win';
  return null;
}
```

### Pattern 3: Responsive Tab Structure (CSS approach)

**What:** Two DOM states — desktop (2 tabs, match list below) and mobile (3 tabs, match list in tab) — controlled by CSS media query, not JS.

**When to use:** Preferred over `useWindowSize` because it avoids JS resize handling and flash-of-wrong-layout.

```jsx
{/* Desktop layout: 2-tab wrapper visible at sm+ */}
<div className="d-none d-sm-block">
  <Tab.Container defaultActiveKey="results">
    <Nav variant="tabs">
      <Nav.Item><Nav.Link eventKey="results">Results</Nav.Link></Nav.Item>
      <Nav.Item><Nav.Link eventKey="standings">Standings</Nav.Link></Nav.Item>
    </Nav>
    <Tab.Content>
      <Tab.Pane eventKey="results"><CrossTable ... /></Tab.Pane>
      <Tab.Pane eventKey="standings"><StandingsContent ... /></Tab.Pane>
    </Tab.Content>
  </Tab.Container>
  {/* Match list always below on desktop */}
  <MatchList matches={matches} ... />
</div>

{/* Mobile layout: 3-tab wrapper visible at xs only */}
<div className="d-sm-none">
  <Tab.Container defaultActiveKey="results">
    <Nav variant="tabs">
      <Nav.Item><Nav.Link eventKey="results">Results</Nav.Link></Nav.Item>
      <Nav.Item><Nav.Link eventKey="standings">Standings</Nav.Link></Nav.Item>
      <Nav.Item><Nav.Link eventKey="matches">Matches</Nav.Link></Nav.Item>
    </Nav>
    <Tab.Content>
      <Tab.Pane eventKey="results"><CrossTable ... /></Tab.Pane>
      <Tab.Pane eventKey="standings"><StandingsContent ... /></Tab.Pane>
      <Tab.Pane eventKey="matches"><MatchList ... /></Tab.Pane>
    </Tab.Content>
  </Tab.Container>
</div>
```

This duplicates the CrossTable and MatchList renders into two DOM trees, but since both are inside `Tab.Pane` which is lazy (Tab.Pane does not render its children until first activated by default in react-bootstrap), the performance cost is acceptable for small group sizes (4-8 players).

**Alternative:** Use a single controlled Tab.Container with a `useWindowSize` / `window.matchMedia` hook to dynamically add/remove the Matches tab. This avoids DOM duplication but requires JS to compute layout. Given project history of CSS-based responsive patterns (see Phase 25 context), CSS approach is preferred.

### Pattern 4: Cross-Highlighting State

**What:** Shared state in `GroupStandingsTable` (or the layout wrapper) that tracks the hovered `matchId`. Both `CrossTable` and `MatchList` read this state and apply highlight class.

**When to use:** Bidirectional highlight requires a common ancestor to hold state.

```javascript
// In GroupStandingsTable
const [highlightedMatchId, setHighlightedMatchId] = useState(null);

// In MatchList row
<div
  onMouseEnter={() => setHighlightedMatchId(match.id)}
  onMouseLeave={() => setHighlightedMatchId(null)}
  className={`... ${highlightedMatchId === match.id ? 'cross-highlight' : ''}`}
>

// In CrossTable cell (finds match in matchLookup)
// Highlight applies to both [row,col] and [col,row] cells
const isHighlighted = match && highlightedMatchId === match.id;
<td
  onMouseEnter={() => match && setHighlightedMatchId(match.id)}
  onMouseLeave={() => setHighlightedMatchId(null)}
  className={isHighlighted ? 'cross-highlight' : ''}
>
```

The CSS class `cross-highlight` should apply a light background (e.g., `background-color: #e8f4fd` — Bootstrap blue-50 shade).

Desktop-only: `highlightedMatchId` state is still maintained on mobile (no harm), but mobile users will never see the visual effect because they can't hover.

### Pattern 5: Modal Invocation from CrossTable Cell Click

**What:** Clicking a cross-table cell invokes `setSelectedMatch(match)` — the same mechanism already used for match list row clicks.

**When to use:** Reuse existing `selectedMatch` state; no new modal state needed.

```javascript
// Empty cell click: find match from lookup, open modal
const handleCellClick = (rowEntityId, colEntityId) => {
  if (rowEntityId === colEntityId) return; // diagonal guard
  const match = matchLookup[rowEntityId]?.[colEntityId]
              || matchLookup[colEntityId]?.[rowEntityId];
  if (match) {
    setSelectedMatch(match);
    // Note: MatchResultModal uses match.player1/player2 order.
    // The "row player = Player 1" requirement means we may need
    // to swap player references when opening — see Pitfall 2 below.
  }
};
```

**Important:** The locked decision "row player = Player 1 in the modal" conflicts with how matches are stored. Matches have a fixed player1/player2 assignment in the DB. If Alice is `player2` in the stored match but Alice is the row player, the modal will show Bob as Player 1 and Alice as Player 2. This means the "row player = Player 1" guarantee requires either (a) finding matches such that the row player is always player1 (depends on how matches are generated), or (b) passing an optional `perspectivePlayerId` prop to MatchResultModal to indicate "display this player as Player 1".

Research finding: Matches in the DB are generated by the group round generation service. The player1/player2 assignment in the match record is fixed at creation time. There is no guarantee that row player is always player1.

**Recommendation for planning:** Add a `swapPerspective` prop to MatchResultModal, or — simpler — just accept that the modal shows players in DB order. The locked decision ("row player = Player 1") may need to be relaxed to "the match opens with both players shown, player1 as stored." The planner should flag this in plans and resolve with the user if needed, or implement a lightweight display-only swap in MatchResultModal.

### Anti-Patterns to Avoid

- **Computing cell scores inline without memoization:** For 6+ player groups this is a 36+ cell render on every state change. Use `useMemo` for `matchLookup`.
- **Storing cross-table as separate state:** The cross-table is fully derived from `matches`. No separate state needed.
- **Triggering SWR re-fetch for symmetric update:** Both cells use the same `matches` SWR data. Calling `mutateMatches()` after result submission already updates both cells because the match object is shared.
- **Using `useWindowSize` hook for responsive tabs:** Causes layout flash. Use Bootstrap `d-sm-none` / `d-none d-sm-block` CSS approach instead (established project pattern from Phase 25).

## Common Pitfalls

### Pitfall 1: Diagonal cell click handler
**What goes wrong:** If `onClick` is applied uniformly to all `<td>` elements including the diagonal, clicking "self" opens a modal with a null match.
**Why it happens:** Diagonal cells have no match in the lookup.
**How to avoid:** Guard with `if (rowEntityId === colEntityId) return;` before any interaction. Remove `cursor: pointer` from diagonal cells. Apply `style={{ cursor: 'default', backgroundColor: '#f0f0f0' }}` to diagonals.
**Warning signs:** Console error on null match access in MatchResultModal.

### Pitfall 2: Player1/Player2 ordering in modal
**What goes wrong:** "Row player = Player 1" contract breaks when the DB stores the match with the column player as player1.
**Why it happens:** Group round generation assigns player1/player2 in a fixed order at draw time, unrelated to the cross-table row/column order.
**How to avoid:** Either (a) accept DB ordering and remove the strict "row player = Player 1" requirement — the modal shows both players regardless, or (b) implement a display-swap: pass the row entity ID to MatchResultModal and swap the displayed names if needed. Score entry is symmetric so this only affects label display.
**Warning signs:** User opens cell [Alice, Bob] but modal header shows "Bob vs Alice".

### Pitfall 3: Symmetric cell update after SWR mutation
**What goes wrong:** Developer expects cells [A,B] and [B,A] to update independently and adds duplicate state.
**Why it happens:** Misunderstanding of how the cross-table derives from `matches`.
**How to avoid:** Both cells derive from the same match object in the `matches` array. Calling `mutateMatches()` triggers a re-fetch; `matchLookup` is recomputed from the new matches data, and both cells re-render automatically. No extra state required.

### Pitfall 4: Cross-table overflow on mobile
**What goes wrong:** A 6-player group generates a 7-column table (1 label + 6 data). At 375px width this overflows.
**Why it happens:** CSS table `min-content` widths exceed viewport.
**How to avoid:** Wrap the cross-table in `<div className="table-responsive">` — Bootstrap's established pattern used in the existing standings table. This adds horizontal scroll on mobile. The "Claude's Discretion" note in CONTEXT.md explicitly delegates this decision.

### Pitfall 5: Highlight state on mobile tab switch
**What goes wrong:** User highlights a cell, then switches to Matches tab on mobile — highlight persists but is invisible, causing confusing state.
**Why it happens:** `highlightedMatchId` is not cleared on tab change.
**How to avoid:** Clear `highlightedMatchId` to `null` on tab `onSelect` event. Or — since mobile has no hover events — this is a non-issue in practice (touch devices don't fire mouseenter/mouseleave).

### Pitfall 6: Doubles initials collision
**What goes wrong:** In doubles, "AB" (Alice Brown) and "AB" (Andrew Baxter) produce identical column headers.
**Why it happens:** Initials are a lossy compression of names.
**How to avoid:** CONTEXT.md delegates this to Claude's Discretion. Recommended approach: append a numeric suffix for collisions (AB, AB2), or use first-name initial + last-name first-3-chars. The planner should document the chosen approach.

### Initials computation
```javascript
// Derives "AL" from "Alice Liddell", "?" if no name
function getInitials(name) {
  if (!name) return '?';
  const parts = name.trim().split(/\s+/);
  if (parts.length >= 2) return (parts[0][0] + parts[parts.length - 1][0]).toUpperCase();
  return parts[0].slice(0, 2).toUpperCase();
}

// Doubles pair initials — Claude's discretion: use first player's initials + "/" + second
function getPairInitials(pairName) {
  // "Alice L / Bob M" → "AL/BM"
  const players = pairName.split(' / ');
  return players.map(p => getInitials(p)).join('/');
}
```

### Match lookup map (useMemo pattern)
```javascript
// Derive inside CrossTable or GroupStandingsTable
const matchLookup = useMemo(() => {
  const map = {};
  if (!matches) return map;
  for (const match of matches) {
    const id1 = isDoubles ? match.pair1?.id : match.player1?.id;
    const id2 = isDoubles ? match.pair2?.id : match.player2?.id;
    if (!id1 || !id2) continue;
    if (!map[id1]) map[id1] = {};
    if (!map[id2]) map[id2] = {};
    map[id1][id2] = { match, rowIsPlayer1: true };
    map[id2][id1] = { match, rowIsPlayer1: false };
  }
  return map;
}, [matches, isDoubles]);
```

Storing `rowIsPlayer1` in the lookup avoids re-computing perspective on every cell render.

## Open Questions

1. **Player1/Player2 ordering in modal**
   - What we know: The locked decision says "row player = Player 1 in the modal." Match records store player1/player2 in DB-creation order, which may differ from row/column order.
   - What's unclear: Whether the group draw generation always assigns player1 consistently, or whether a display-swap is needed in MatchResultModal.
   - Recommendation: Planner should add a task to check if match generation preserves row-player=player1. If not, implement a lightweight name-swap prop in MatchResultModal (display-only, no effect on score submission logic).

2. **Doubles initials collision handling**
   - What we know: CONTEXT.md marks this as Claude's Discretion. Pairs in a group are typically 4-8, so collisions are unlikely but possible.
   - What's unclear: Whether to use pair initials ("AL/BM") or combined ("ALBM") for column headers.
   - Recommendation: Use "AL/BM" format (readability) with numeric suffix if collision detected.

3. **matchNumber field presence in useMatches() response**
   - What we know: The Prisma schema has `matchNumber Int` on the Match model. The `getMatches()` API fetches from `/v1/tournaments/:id/matches`.
   - What's unclear: Whether the API response includes `matchNumber` in the serialized match objects (not verified in matchController.js or response serialization).
   - Recommendation: Planner should add a verification step to confirm `match.matchNumber` is present in the API response before displaying it in the match list.
