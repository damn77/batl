## Decisions Made

- **DRY_RUN_RESULT throw pattern:** The service throws a custom error with `code='DRY_RUN_RESULT'` and `data=impactSummary` inside the Prisma transaction. This causes automatic rollback. The controller catches this specific code and returns 200 with the impact data. This avoids needing a separate dry-run code path outside the transaction.
- **Score-only passthrough:** When `previousResultJson` exists and the winner hasn't changed, `submitResult` returns immediately after `tx.match.update` — skipping advanceBracketSlot, consolation routing, and cascade entirely. This is correct: score corrections should have no bracket side effects.
- **New exports from consolationEligibilityService:** `cascadeClearMainBracket` (full implementation) and `cascadeClearConsolationDownstream` (thin wrapper around existing private `cascadeClearWinnerFromNextRounds`). Avoids duplication while exposing what matchResultService needs.
- **Read-only traversal helpers:** `findDownstreamMainMatches`, `findDownstreamConsolationMatches`, and `findDownstreamConsolationMatchesFromMatch` are private to matchResultService and only used by the dry-run path. They mirror the cascade algorithm but only read.
- **Test mock strategy:** Used `jest.unstable_mockModule('@prisma/client')` with a stateful in-memory array mock implementing `findUnique/findFirst/findMany/update/upsert` with a simple recursive where-clause matcher. This allows real service logic to run against controlled DB state.
