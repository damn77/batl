**Step 1 â€” Non-organizer winner-change block (BB-01):**
After the existing organizer-lock check (line ~132), add a new check: if `previousResultJson` exists AND `!isOrganizer`, parse the previous winner and compare with `body.winner`. If they differ, `throwError(403, 'WINNER_CHANGE_NOT_ALLOWED', 'Only a tournament organizer can change the winner of a completed match. You can update the score without changing the winner.')`.

**Step 2 â€” Score-only passthrough (BB-02):**
After writing the result (existing `tx.match.update`), detect whether the winner actually changed. If `previousResultJson` exists and the previous winner string === the new winner string, skip ALL downstream logic (no advanceBracketSlot, no consolation routing, no cascade). Just return the updated match. This short-circuits the entire cascade path for score-only corrections.

**Step 3 â€” Export cascadeClearMainBracket from consolationEligibilityService.js:**
Create a new exported async function `cascadeClearMainBracket(tx, bracketId, sourceMatchId, winnerId, winnerIsDoubles)` that replicates the `cascadeClearWinnerFromNextRounds` pattern but for main bracket matches. The algorithm:
1. Find sourceMatch's round number
2. Find winnerId in any match in the same bracket at a HIGHER round number (using the same OR pattern for player/pair slots)
3. If found: determine if that match also produced a further winner who advanced
4. Clear winnerId's slot in that match, reset result/status/completedAt to null/SCHEDULED/null, set isBye to false
5. Recurse on the further winner if one exists
6. Return an array of all cleared match objects (for dry-run reporting)

**Step 4 â€” Organizer winner-change cascade (BB-03, BB-04):**
In submitResult, after writing the new result, when a winner change IS detected (previousWinnerStr !== newWinnerStr):

a) **Clear old winner from downstream main bracket** â€” call `cascadeClearMainBracket(tx, updated.bracketId, updated.id, oldWinnerId, isDoubles)`. The old winner was advanced into the next main bracket match; this clears them recursively.

b) **Advance new winner** â€” call `advanceBracketSlot(tx, updated, newWinnerId)` to place the new winner in the next main bracket slot.

c) **Clear old loser from consolation** â€” existing `clearConsolationRouting(tx, tournamentId, oldLoserId, isDoubles)` already handles this (already in the code at lines 201-218). Keep this existing logic.

d) **Route new loser to consolation** â€” existing `routeLoserToConsolation(tx, updated, newWinnerId, isOrganizer)` already handles this (already in code at line 222). Keep this existing logic.

e) **BYE check** â€” existing `checkBYEWinnerConsolationUpdate(tx, updated, newWinnerId)` already handles this (line 226). Keep this.

The key change is inserting step (a) BEFORE the existing advanceBracketSlot call, and making the existing advanceBracketSlot call conditional on winner change: if winner changed, the old winner's slot in the next match must first be cleared (step a), then the new winner placed (step b). If winner did NOT change (score-only), skip all of this per Step 2.

**Step 5 â€” Consolation bracket winner-change cascade (BB-05):**
When the completed match is in a CONSOLATION bracket and the winner changes, call `cascadeClearWinnerFromNextRounds` (which already exists as module-private in consolationEligibilityService.js) on the old winner. To enable this from matchResultService, either:
- Export it as `cascadeClearConsolationDownstream`, OR
- Create a thin wrapper export that calls it

Then in submitResult, detect bracket type: if the match's bracket is CONSOLATION and winner changed, call the consolation cascade on the old winner before advancing the new winner.

**Step 6 â€” Dry-run mode (BB-04):**
When `dryRun === true`, instead of performing the cascade, detect impacted downstream matches and return an impact summary without modifying data.

Inside the transaction, after the winner-change detection:
1. Call a new helper `getDownstreamImpact(tx, bracketId, matchId, winnerId, isDoubles)` that traverses the same path as cascade-clear but only READS matches (counts them, collects player names from their player1/player2/pair1/pair2 relations).
2. For main bracket matches, also check consolation bracket for impacted matches (old loser may have consolation results).
3. Count total impacted matches (main + consolation) and collect affected player/pair names.
4. If dryRun is true, roll back the transaction (throw a special non-error object or return early) and return the impact data.

Implementation approach: when dryRun=true, run the entire function up to the winner-change detection point, compute impact, then throw a custom "DRY_RUN_RESULT" that the controller catches and returns as a 200 response. The transaction auto-rolls back on throw.

Shape of dry-run response:
```json
{
  "dryRun": true,
  "impactedMainMatches": 2,
  "impactedConsolationMatches": 1,
  "totalImpactedMatches": 3,
  "affectedPlayers": ["John Doe", "Jane Smith"],
  "requiresConfirmation": true
}
```

When totalImpactedMatches === 0, set `requiresConfirmation: false`.

**Step 7 â€” BYE auto-advancement verification (BB-06):**
This is already handled by existing Phase 5 code (`routeLoserToConsolation` step 11, `checkBYEWinnerConsolationUpdate`). The cascade-clear in Step 4 ensures that when a winner changes, consolation routing is corrected. No new code needed beyond what's already there + the cascade. Verify by reading through the flow: when main bracket M1 result changes winner, old loser cleared from consolation (step 4c), new loser routed (step 4d), BYE check runs (step 4e). P1/P2 positions are determined by `posInRound % 2 === 0` in routeLoserToConsolation step 8.
