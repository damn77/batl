  <action>
Add post-placement opponent advancement to `recordOptOut()` in `consolationOptOutService.js`.

Import `advanceBracketSlot` from `./tournamentLifecycleService.js` and `PrismaClient` is already used as top-level `prisma` instance.

Wrap the existing Step 6 (create opt-out) and the new Step 7 (advance opponent) in a `prisma.$transaction()`. The consolationBracket is already fetched in Step 5 — use its id in the transaction.

**New Step 7 logic (add after opt-out create, inside the transaction):**

1. Find the consolation match where the opted-out player/pair currently occupies a slot. Use the same `participantFilter` built in Step 5 (the one used for `playedConsolationMatches` query). Query: `tx.match.findFirst({ where: { bracketId: consolationBracket.id, ...participantFilter, status: { not: 'COMPLETED' } }, select: { id, roundId, bracketId, matchNumber, tournamentId, isBye, result, status, player1Id, player2Id, pair1Id, pair2Id } })`. If no match found (player not yet placed — pre-placement opt-out), skip advancement.

2. Determine which slot the opted-out entity occupies and who the opponent is:
   - For singles (playerId): if match.player1Id === playerId → opponentId = match.player2Id, opponentSlot = 'player2'; else opponentId = match.player1Id.
   - For doubles (pairId): if match.pair1Id === pairId → opponentId = match.pair2Id, else opponentId = match.pair1Id.

3. If opponentId is null (opponent not yet placed), no advancement needed — opt-out is recorded but match will be handled by routeLoserToConsolation's existing auto-BYE logic when the opponent eventually arrives.

4. If opponentId is not null, call `advanceBracketSlot(tx, currentMatch, opponentId)`. This advances the opponent as if they won a BYE/forfeit.

The entire operation (opt-out create + advancement) must be atomic inside `prisma.$transaction()`. Move the `consolationOptOut.create` into the transaction callback, using the transaction client `tx` instead of `prisma`. Keep the pre-transaction queries (Steps 1-5) outside the transaction as they are read-only guards.

Do NOT use `prisma` directly inside the transaction callback — use the `tx` parameter.
